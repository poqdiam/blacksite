Roles & Permissions

Every user on the platform is assigned a role that controls what they can see and do. Roles are set by administrators in the User Management section. The table below summarizes each role and its primary capabilities.

Admin Platform administrator Full access to all features, user management, system configuration, SIEM, audit log
AO Authorizing Official — final ATOs decision-maker ATO submissions, decision queue, posture overview, risk review
CISO Chief Information Security Officer Program-wide posture, risk, controls, submissions, reports
ISSM IS Security Manager — org-level program Full GRC operations, SIEM, audit log, observations; no user management
ISSO IS Security Officer — system-level System controls, POA&M, risks, observations, ATO package, RMF
SCA Security Control Assessor — independent assessment Controls review, SSP upload, POA&M review, observations, posture

{# ── Account Status ── #}

Account Status

Each user account has one of three states:

Active Normal access

The user can log in and access all features permitted by their role.

Frozen Temporarily suspended

The account is suspended. The user receives a 403 Access Denied on every page. Their profile, role, and system assignments are preserved. An admin can unfreeze the account instantly, restoring full access with no data loss.

Removed Deactivated — archived for 365 days

The account is deactivated and the user cannot log in. The profile is moved to the Shadow Archive and automatically purged after 365 days. An admin can restore a removed account within that window. Removal requires selecting a reason (e.g., Termination, Security Violation) which is recorded in the audit log.

{# ── Freezing ── #}

Freezing an Account

Use Freeze when you need to temporarily revoke a user's access without permanently removing their account. Common scenarios:

  • User is on extended leave and access should be suspended
  • Investigation is underway — access revoked pending outcome
  • User's role change is pending approval
  • Account compromise suspected — immediate lockout required
1
Go to Admin → User Management.
2
Find the user in the table. Click the Freeze button in their Actions column.
3
The account is immediately suspended. The user row shows the frozen badge. The action is recorded in the audit log.
4
To restore access, click Unfreeze. Access is restored immediately — no data is lost.
Note: Frozen accounts retain their role and system assignments. The freeze only affects login access — all GRC data associated with the user remains intact.
{# ── Removing ── #}

Removing a User

Use Remove for permanent account deactivation, typically at offboarding. Removal is a two-step process designed to prevent accidental deletions.

1
Go to Admin → User Management. Click Remove on the user's row.
2
A confirmation screen shows the user's username, role, and number of active system assignments.
3
Select a Removal Reason from the dropdown:
  • Termination — Employment ended
  • Role Elimination — Position no longer exists
  • Security Violation — Policy or security breach
  • Contractor Offboarding — Contract concluded
  • Account Compromise — Credentials believed compromised
4
Click Yes, Remove User. The account moves to the Shadow Archive and the reason is recorded in the audit log.
5
Within 365 days, the account can be Restored from Admin → Shadow Archive. After 365 days, the profile is automatically purged.
System Assignments: When a user is removed, their system assignments are not deleted — they remain for audit continuity. The user simply loses the ability to log in. If the assignments should be reassigned, do so before or after removal via the System Catalog.
{# ── Role Switching ── #}

Role Switching

Admins and senior roles (AO, CISO, ISSM) can temporarily switch to a lower-privilege role to verify how the platform appears to users in that role. This is useful for access reviews and troubleshooting permission issues.

  • Click your username in the top-right corner to open the user menu.
  • Use the Switch Role dropdown to select a role.
  • An orange banner appears at the top of every page: VIEWING AS [ROLE]
  • While in a switched role, you only see what that role sees — admin menus and restricted pages are inaccessible.
  • Click ← Reset Role in the banner or user menu to return to your native role.
Note: Role switching is session-based (stored in a browser cookie). Closing your browser or clicking Reset Role ends the switched view. No data is modified while viewing as another role.
{# ── POA&M ── #}

POA&M Tracker

The Plan of Action & Milestones (POA&M) tracker follows the NIST SP 800-53 CA-5 format. Each item represents a security weakness that requires remediation tracking.

Status Lifecycle

POA&M items flow through the following states:

Draft → Open → In Progress → Blocked → Ready for Review

After review, items close in one of four terminal states:

Closed Verified Deferred by Waiver Accepted Risk False Positive
StatusMeaning
DraftItem created but not yet formally opened — may be incomplete
OpenWeakness confirmed and formally tracked; remediation not yet started
In ProgressRemediation actively underway
BlockedRemediation halted — awaiting resources, approvals, or dependencies
Ready for ReviewRemediation complete; awaiting ISSM/AO verification
Closed VerifiedControl assessor confirmed the weakness is fully remediated
Deferred by WaiverAO-approved waiver granted; item deferred with expiry date
Accepted RiskRisk formally accepted by AO; no further remediation planned
False PositiveFinding determined to be invalid; item closed without remediation
{# ── RMF Workflow ── #}

RMF Workflow

The Risk Management Framework (RMF) workflow follows NIST SP 800-37 Rev 2. It structures how systems are authorized to operate, progressing through six steps:

1
Prepare — Establish organizational risk management roles, strategies, and risk tolerance. Define the system boundary and mission/business processes.
2
Categorize — Classify the system and its information using FIPS 199 / NIST SP 800-60. Assign a security categorization (Low, Moderate, High) for Confidentiality, Integrity, and Availability.
3
Select — Choose an appropriate control baseline (NIST SP 800-53) aligned to the security categorization. Tailor controls as needed.
4
Implement — Deploy the selected controls. Document implementation details in the System Security Plan (SSP).
5
Assess — SCA independently evaluates control effectiveness. Findings feed the Security Assessment Report (SAR) and open POA&M items.
6
Authorize — AO reviews the authorization package (SSP + SAR + POA&M) and issues an Authorization to Operate (ATO), Interim ATO (IATO), or Denial of ATO (DATO).
★
Monitor — Continuous monitoring of controls, security alerts, and POA&M progress throughout the ATO lifecycle.
{# ── Control Catalog ── #}

Control Catalog

The Control Catalog displays all security controls from NIST SP 800-53 Rev 5, organized by control family (AC — Access Control, AU — Audit, etc.). Each control page shows:

  • Statement — The normative control requirement text from NIST
  • Supplemental Guidance — Additional context from NIST on intent and implementation
  • Family — The control family (e.g., AC, AU, IA)
  • Parameters — Configurable values within the control (shown where applicable)

Use the filter on the Controls list page to search by control ID, title, or family. Controls cannot be edited from the catalog view — they are loaded from the OSCAL catalog source.

{# ── ATO Package ── #}

ATO Package

The ATO Package section manages all authorization artifacts for a system. Document types include the System Security Plan (SSP), Security Assessment Report (SAR), Plan of Action & Milestones (POA&M), and 52 additional document types aligned to federal authorization requirements.

  • Navigate to ATO Package → select a system to see all required documents
  • Upload or submit documents by clicking the document type card
  • Document status tracks through: Not Started → In Progress → Under Review → Approved
  • The ATO Submissions page tracks submitted authorization packages for AO review

The ATO Package supports EIS (Enterprise Infrastructure Solutions) and standard authorization types, including FedRAMP Moderate, FedRAMP High, and FISMA equivalents.

{# ── Observations ── #}

Observations

Observations are formal security findings recorded during assessments, audits, or continuous monitoring. They are linked to systems and controls and can feed into POA&M items.

  • The sidebar badge on Observations shows the count of open, unresolved observations
  • Each observation has a severity (Low, Moderate, High, Critical), a finding statement, and an optional control reference
  • ISSOs, ISSMs, SCAs, and Auditors can create and manage observations
  • The /api/observations/summary endpoint provides a JSON summary for dashboard widgets
{# ── SIEM & Audit Log ── #}

SIEM & Audit Log

SIEM (Security Information & Event Management)

The built-in SIEM at Admin → SIEM logs platform security events automatically. Events are captured when:

  • Any request returns HTTP 400 or higher (errors, denied access)
  • Any /admin/* page is accessed
  • Login, auth, role-switching, or shell-related paths are accessed

Severity levels:

  • Critical / High — Server errors (5xx), critical events
  • Medium — Failed auth (401), access denied (403)
  • Low — Admin page access
  • Info — Other logged events

Audit Log

The Audit Log at Admin → Audit Log records all data mutations in the platform: user role changes, system creation, POA&M updates, ATO submissions, freezes, removals, etc. Each entry captures the performing user, timestamp, action type, resource, and details.

  • Default view hides VIEW events (read-only accesses) to reduce noise
  • Toggle "Include View Events" to see all access records
  • Default date range is the last 30 days
